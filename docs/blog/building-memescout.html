Building a Production-Ready Meme Coin Tracker with NestJS

Pratik Fulkar February 15, 2026 8 min read
NestJS TypeScript PostgreSQL Prisma Docker

The meme coin market is explosive, unpredictable, and full of opportunity. Building a robust platform to track and analyze these volatile assets requires careful architectural decisions, scalable infrastructure, and production-grade code. In this post, I'll walk you through how we built MemeScout‚Äîa comprehensive backend API for discovering and tracking meme coins.

Why We Built MemeScout

The crypto space moves fast, and meme coins move even faster. Investors and traders need real-time data, historical insights, and intelligent scoring to make informed decisions. We identified several key requirements:

  • Real-time discovery of new meme coins from DEX platforms
  • Historical price tracking with snapshots
  • Intelligent scoring algorithms based on multiple factors
  • High availability and performance for production workloads
  • Direct integration with CoinMarketCap for seamless data access

Architecture Overview

We chose a modern, production-ready tech stack that prioritizes type safety, developer experience, and scalability:

NestJS 10

A progressive Node.js framework that brings structure and best practices to our API. NestJS provides excellent TypeScript support, dependency injection, and a modular architecture that scales beautifully.

PostgreSQL 15

Our primary database for storing tokens, price snapshots, user watchlists, and alerts. PostgreSQL offers ACID compliance, advanced indexing, and excellent performance for complex queries.

Prisma ORM

Type-safe database access with auto-generated TypeScript types. Prisma's migration system and query builder make database operations a breeze while maintaining full type safety.

Redis

High-performance caching layer to reduce database load and improve API response times. Essential for frequently accessed data like token lists and price snapshots.

Key Features Implementation

1. New Launches Tracker

One of our most powerful features is the new launches tracker. It identifies newly launched meme coins with high trading volume, providing investors with early opportunities. Here's how we implemented it: 

async getNewlyLaunchedCoins(
  maxDaysSinceLaunch: number = 30,
  minVolume24h: number = 100000,
  limit: number = 50,
): Promise<NewLaunchesResponseDto> {
  const cutoffDate = new Date();
  cutoffDate.setDate(cutoffDate.getDate() - maxDaysSinceLaunch);

  const tokens = await this.prisma.token.findMany({
    where: {
      launchDate: { gte: cutoffDate },
      volume24h: { gte: minVolume24h },
    },
    orderBy: [
      { volume24h: 'desc' },
      { launchDate: 'desc' },
    ],
    take: limit,
    include: {
      chain: true,
      snapshots: {
        orderBy: { timestamp: 'asc' },
        take: 100,
      },
    },
  });

  // Calculate price changes and format response...
}

This query efficiently filters tokens by launch date and volume, orders them by relevance, and includes historical snapshots for price analysis. Database indexes on launchDate and volume24h ensure fast query performance.

2. CoinMarketCap Integration

Every token in our system includes a clickable CoinMarketCap URL, making it seamless for users to access detailed market data and charts. The implementation is straightforward but powerful: 

generateCoinbaseUrl(symbol: string): string {
  const normalizedSymbol = symbol.toUpperCase();
  return `https://coinmarketcap.com/search?q=${normalizedSymbol}`;
}

This simple function generates consistent URLs that our frontend can use to create clickable links. All endpoints automatically include this URL in their responses.

3. Price History & Snapshots

Historical price data is crucial for trend analysis. We use a snapshot system that records price, volume, and meme score at regular intervals: 

async getTokenPriceHistory(tokenId: string, days: number = 30) {
  const cutoffDate = new Date();
  cutoffDate.setDate(cutoffDate.getDate() - days);

  const snapshots = await this.prisma.tokenSnapshot.findMany({
    where: {
      tokenId,
      timestamp: { gte: cutoffDate },
    },
    orderBy: { timestamp: 'asc' },
  });

  return snapshots.map((snapshot) => ({
    priceUsd: snapshot.priceUsd,
    volume24h: snapshot.volume24h,
    memeScore: snapshot.memeScore,
    timestamp: snapshot.timestamp,
  }));
}

Database Schema Design

Our Prisma schema is designed for performance and scalability. Key design decisions include:

  • Strategic indexes on frequently queried fields (memeScore, launchDate, volume24h)
  • Timestamp-based partitioning for price snapshots
  • Normalized relationships between tokens, chains, watchlists, and users
  • Optional fields for backward compatibility during migrations

API Design & Documentation

We use Swagger/OpenAPI for comprehensive API documentation. Every endpoint is documented with:

  • Request/response schemas with TypeScript types
  • Query parameter descriptions and defaults
  • Example requests and responses
  • Error codes and handling

This makes it easy for frontend developers and API consumers to integrate with MemeScout.

Performance Optimization

To handle high traffic and provide fast responses, we implemented several optimizations:

Deployment & DevOps

MemeScout is containerized with Docker and can be deployed anywhere that supports Docker containers. Our docker-compose.yml includes:

  • PostgreSQL database with persistent volume
  • Redis cache
  • The NestJS application
  • Network configuration for service communication

docker-compose up -d

# Run database migrations
npm run prisma:migrate

# Access API at http://localhost:3000
# Swagger docs at http://localhost:3000/docs

Lessons Learned

Building MemeScout taught us valuable lessons about production-grade API development:

  1. Type safety is invaluable - TypeScript + Prisma caught countless bugs before they reached production
  2. Start with good architecture - NestJS's modular structure made it easy to add new features
  3. Performance matters from day one - Indexes and caching are easier to add early than later
  4. Documentation drives adoption - Swagger made it easy for others to use our API
  5. Real-world data is messy - Robust error handling and data validation are essential
